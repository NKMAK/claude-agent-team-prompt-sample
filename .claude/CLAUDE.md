# プロジェクト方針

## 設計フェーズのワークフロー

ソフトウェアの設計作業を依頼されたとき、以下のスキルを活用して成果物を作成する。

### 利用可能なスキル

| スキル | 用途 | 呼び出し |
|--------|------|----------|
| 要件定義 | ユーザーの要望を整理し、要件ドキュメントを作成 | `/requirements-gathering` |
| タスク計画 | 要件をタスクに分解し、実装計画を作成 | `/task-planning` |
| アーキテクチャ設計 | ファイル構成と技術設計を作成 | `/architecture-design` |
| 設計レビュー | 計画と設計の品質を検証 | `/design-review` |

### 基本フロー

1. **要件定義**: ユーザーにヒアリングし、要件を明確化する（`/requirements-gathering`）
2. **計画 + 設計**: 要件が固まったら、タスク計画とアーキテクチャ設計を作成する（`/task-planning` + `/architecture-design`）
3. **レビュー**: 計画・設計を検証する（`/design-review`）

### エージェントチームで使う場合

Agent Teamsで設計フェーズを実行する場合、リーダーは以下のように運用する：

1. まず自身が `/requirements-gathering` を参照してユーザーから要件をヒアリングする
2. 要件が固まったら、チームメイトにタスクを割り当てる：
   - 計画担当: `/task-planning` スキルを参照してタスク計画を作成
   - 設計担当: `/architecture-design` スキルを参照してアーキテクチャ設計を作成
3. 成果物が出たら、リーダーまたはレビュー担当が `/design-review` を参照してレビューする
4. 問題があればユーザーに確認し、修正を指示する

### 成果物の保存先

```
.claude/design-output/[project-name]/
├── requirement-doc.md
├── plan-doc.md
├── architecture-doc.md
└── review-doc.md
```

## 設計の基本原則

- **YAGNI**: 今必要な機能だけを作る。将来のための機能は含めない
- **1ファイル1責任**: ファイルサイズは200-300行以内を目安
- **ボトムアップ開発**: 基盤→ビジネスロジック→UI の順で計画する
- **シンプルさ優先**: 過度な抽象化を避け、最もシンプルな解決策を選ぶ
